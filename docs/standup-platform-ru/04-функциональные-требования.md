# 04. Функциональные требования

Ниже перечислены требования уровня “что система обязана уметь”. Нумерация нужна для декомпозиции задач и последующей трассировки в backlog/QA.

## 4.1. Identity, профиль, роли

### AUTH

- `AUTH-001` Система должна поддерживать вход через Telegram.
- `AUTH-002` Система должна поддерживать вход через VK.
- `AUTH-003` Система должна поддерживать вход через Google.
- `AUTH-004` Для iOS-релиза система должна поддерживать `Sign in with Apple`.
- `AUTH-005` Система должна создавать единый внутренний user identity независимо от внешнего провайдера.
- `AUTH-006` Пользователь должен иметь возможность привязать несколько auth providers к одному аккаунту.
- `AUTH-007` Сессия должна восстанавливаться автоматически при валидных токенах.
- `AUTH-008` Пользователь должен иметь возможность выйти со всех устройств.

### PROFILE

- `PROFILE-001` Пользователь должен хранить базовый профиль: имя, username/handle, фото, город.
- `PROFILE-002` Пользователь должен видеть свои доступные роли.
- `PROFILE-003` Пользователь должен переключать активный контекст роли без re-login.
- `PROFILE-004` Комик должен иметь отдельные поля профиля: bio, теги, длительность сета, ссылки, payout status.

## 4.2. Organizer workspace

- `ORG-001` Организатор должен иметь workspace, отделённый от личного профиля.
- `ORG-002` Workspace должен поддерживать несколько сотрудников.
- `ORG-003` Owner должен приглашать сотрудников по ссылке или user identity.
- `ORG-004` Система должна поддерживать роли `owner`, `manager`, `checker`, `host`.
- `ORG-005` Все критичные действия сотрудников должны попадать в audit log.

## 4.3. Площадки и конструктор зала

- `VENUE-001` Организатор должен создавать площадки и хранить их каталог.
- `VENUE-002` Площадка должна хранить адрес, название, описание, вместимость, таймзону, контакты.
- `VENUE-003` Система должна поддерживать шаблоны схем зала.
- `VENUE-004` Конструктор зала должен поддерживать:
  - сцену;
  - сектора;
  - ряды;
  - отдельные места;
  - столы;
  - технические зоны;
  - blocked seats;
  - ценовые зоны.
- `VENUE-005` Одну схему зала можно использовать для нескольких мероприятий.
- `VENUE-006` Организатор должен иметь возможность локально переопределять цены и доступность мест для конкретного события.
- `VENUE-007` Система должна хранить версию схемы зала, чтобы старые заказы не ломались после редактирования template.

## 4.4. Мероприятия

- `EVENT-001` Организатор должен создавать событие с датой, временем, площадкой и описанием.
- `EVENT-002` Система должна поддерживать статусы события.
- `EVENT-003` Событие должно иметь публичную карточку.
- `EVENT-004` Организатор должен управлять расписанием продаж: открыть, приостановить, закрыть.
- `EVENT-005` Организатор должен иметь возможность отменить мероприятие.
- `EVENT-006` Организатор должен назначать ответственных за зону ответственности:
  - check-in;
  - сцена;
  - гости;
  - касса/продажи.
- `EVENT-007` Система должна поддерживать расписание сборов и внутренних заметок команды.

## 4.5. Каталог и карточка мероприятия

- `CATALOG-001` Зритель должен видеть каталог мероприятий.
- `CATALOG-002` Каталог должен фильтроваться по городу, дате, площадке, цене и статусу.
- `CATALOG-003` Карточка мероприятия должна показывать:
  - описание;
  - дату и время;
  - площадку;
  - схему зала или доступные тарифы;
  - лайнап;
  - наличие мест;
  - правила возврата.
- `CATALOG-004` Карточка мероприятия должна поддерживать deeplink.

## 4.6. Заявки комиков и лайнап

- `LINEUP-001` Комик должен иметь возможность подать заявку на мероприятие.
- `LINEUP-002` Организатор должен видеть список заявок и фильтровать его.
- `LINEUP-003` Организатор должен менять статус заявки.
- `LINEUP-004` Approved заявки должны попадать в lineup.
- `LINEUP-005` Лайнап должен иметь явный order index.
- `LINEUP-006` Организатор или host должен менять порядок выступлений до и во время события.
- `LINEUP-007` Комик должен видеть только релевантную ему operational информацию.
- `LINEUP-008` Система должна поддерживать live-status:
  - `up_next`;
  - `on_stage`;
  - `done`;
  - `delayed`;
  - `dropped`.
- `LINEUP-009` Изменения live-status должны приходить в real time.

## 4.7. Билеты, инвентарь и checkout

- `TICKET-001` Пользователь должен выбирать конкретное место или тарифную зону.
- `TICKET-002` Система должна создавать hold мест на ограниченное время.
- `TICKET-003` Hold должен автоматически истекать и освобождать место.
- `TICKET-004` Один seat inventory unit не может быть продан дважды.
- `TICKET-005` Заказ должен иметь отдельный order lifecycle.
- `TICKET-006` У заказа должен быть idempotency key для защиты от дублей оплаты.
- `TICKET-007` После успешной оплаты должен выпускаться билет с QR.
- `TICKET-008` Организатор должен иметь возможность выписать complimentary ticket.
- `TICKET-009` Пользователь должен видеть историю заказов и билетов.
- `TICKET-010` Система должна поддерживать возврат или аннулирование билета.
- `TICKET-011` Система должна уметь возвращать место обратно в продажу после refund/cancel.
- `TICKET-012` Для sold out событий должен быть waitlist.

## 4.8. Check-in

- `CHECKIN-001` Checker должен сканировать QR-код билета.
- `CHECKIN-002` Система должна сразу показывать валидность билета.
- `CHECKIN-003` Повторное сканирование должно быть явно помечено как duplicate/reuse.
- `CHECKIN-004` Check-in должен логироваться.
- `CHECKIN-005` Приложение checker должно уметь кратковременно работать при нестабильной сети.

## 4.9. Донаты

- `DONATION-001` Зритель должен иметь возможность открыть экран доната из карточки комика и из live-экрана события.
- `DONATION-002` Система должна поддерживать preset amounts и произвольную сумму.
- `DONATION-003` Зритель может добавлять короткое сообщение к донату.
- `DONATION-004` Донат должен иметь собственный payment lifecycle.
- `DONATION-005` Комик должен видеть историю и сумму донатов.
- `DONATION-006` Донаты разрешены только для verified payout profiles.
- `DONATION-007` Система должна уметь приостанавливать донаты при проблемах комплаенса или payout.
- `DONATION-008` Донат не должен блокировать покупку билета и жить как самостоятельный поток.

## 4.10. Уведомления и коммуникация

- `NOTIFY-001` Пользователь должен получать push по критичным событиям.
- `NOTIFY-002` Комик должен получать push по статусу заявки и изменениям лайнапа.
- `NOTIFY-003` Зритель должен получать push по покупке, изменению времени, отмене и возврату.
- `NOTIFY-004` Организатор должен получать push по продажам, ошибкам оплаты и инцидентам.
- `NOTIFY-005` MVP должен включать event feed/announcements; полноценный общий чат можно отложить.

## 4.11. Администрирование и поддержка

- `ADMIN-001` Система должна поддерживать просмотр инцидентов по заказам и платежам.
- `ADMIN-002` Система должна вести audit log по критичным действиям.
- `ADMIN-003` Система должна поддерживать ручное исправление проблемного заказа через backoffice/API.
- `ADMIN-004` Система должна поддерживать мягкую блокировку мошеннического пользователя или payout profile.

## 4.12. Аналитика

- `ANALYTICS-001` Организатор должен видеть продажи по мероприятию.
- `ANALYTICS-002` Организатор должен видеть заполненность зала и sold/held/blocked inventory.
- `ANALYTICS-003` Организатор должен видеть статистику заявок комиков.
- `ANALYTICS-004` Комик должен видеть число выступлений и донаты.
- `ANALYTICS-005` Продукт должен собирать funnel-метрики:
  - просмотр карточки;
  - выбор места;
  - начало checkout;
  - успешная оплата;
  - check-in;
  - донат.

## 4.13. Что считаем обязательным качеством готовности

Функция не считается завершённой, если отсутствуют:
- доменные инварианты;
- негативные кейсы;
- события для аналитики и журналирования;
- тесты на основной happy path и хотя бы один edge case.
