# 11. Статус реализации на 2026-03-10

Этот файл синхронизирует target-state спецификацию из `docs/standup-platform-ru/` с фактическим состоянием репозитория.

## Как интерпретировать пакет

- Файлы `01`-`10` описывают целевой продукт и рекомендуемую архитектуру.
- Этот файл фиксирует, что уже есть в коде, что частично подготовлено, а что пока остаётся планом.
- При конфликте между этим статусом и кодом сначала проверять код и `docs/context/*`, затем обновлять статус.

## Текущий срез репозитория

### Реализовано

- `Kotlin Multiplatform` mobile foundation:
  - `composeApp`,
  - `iosApp`,
  - `shared`,
  - `core/common`.
- Shared auth/session foundation:
  - `feature:auth`,
  - `data:auth`,
  - `shared` session bridges/state.
- Backend auth/session foundation на `Ktor`:
  - Telegram verify endpoint,
  - `session/me`,
  - `refresh`,
  - `logout`,
  - `me/roles`,
  - `me/active-role`,
  - `workspaces` create/list,
  - PostgreSQL persistence,
  - refresh-token rotation,
  - replay/rate-limit/body-size/request-id hardening.
- Provider-agnostic backend identity foundation:
  - internal `User`,
  - linked auth identities,
  - role storage,
  - active role context,
  - organizer workspace owner membership.
- Android root navigation with feature-owned auth graph.
- iOS root graph container with auth/main graph shells.
- OpenAPI contract и CI/CD/deploy stack для текущего server scope.

### Частично готово

- VK auth и Google auth заведены в mobile data layer, но пока работают как client-side stubs без реального backend exchange и unified provider linking.
- Shared session model существует, но роли, active-role context и organizer workspace membership ещё не смоделированы.
- Mobile shared/session layer и UI пока не используют новые backend roles/workspace endpoints.
- iOS navigation/bridge foundation есть, но не закрывает целевые продуктовые домены beyond auth/main placeholder.

### Не реализовано

- `Sign in with Apple`.
- Multi-role profile model, provider linking, role switching UI.
- Organizer workspace staff invitations and full member management.
- Venues, hall templates, event snapshots.
- Events, sales lifecycle, staff assignments.
- Ticketing, holds, orders, refunds, QR check-in.
- Comedian applications, lineup, live stage status.
- Donations, payout verification, payout ledger.
- Notifications/feed, analytics, backoffice/admin tooling.

## Синхронизация по документам

### 01. Видение и границы продукта

- Актуально как target-state.
- Репозиторий находится на раннем foundation-этапе и пока покрывает только auth/session slice из будущего MVP.

### 02. Исследование аналогов

- Актуально как продуктовая рамка.
- Практические выводы уже зафиксированы в контексте и ADR `D-045`-`D-048`, но их реализация в коде почти не началась за пределами auth foundation.

### 03. Роли и сценарии

- Модель “один аккаунт, несколько ролей” принята как продуктовое решение.
- Фактически в коде есть только базовый internal user/session для auth.
- Role assignments, active role context и owner-level workspace membership уже заложены на backend.
- Organizer subroles beyond owner, invites, payout status и полные жизненные циклы сценариев пока не реализованы.

### 04. Функциональные требования

- Частично закрыты:
  - `AUTH-001` Telegram auth;
  - `AUTH-007` auto session restore;
  - `AUTH-008` sign out via server session revocation.
- Частично подготовлены, но не доведены до production-ready:
  - `AUTH-002` VK;
  - `AUTH-003` Google.
- Не начаты:
  - `AUTH-004` Apple;
  - `AUTH-005` unified linked identity across providers;
  - `AUTH-006` provider linking;
  - почти весь остальной функциональный scope `ORG`, `VENUE`, `EVENT`, `LINEUP`, `TICKET`, `CHECKIN`, `DONATION`, `NOTIFY`, `ADMIN`, `ANALYTICS`.

### 05. Архитектура системы

- Совпадает по базовому направлению:
  - `KMP` mobile;
  - Compose + SwiftUI split;
  - `Ktor` backend;
  - `PostgreSQL`;
  - `Redis` в стеке для auth rate limiting.
- В репозитории реально выделены только foundation-модули:
  - `feature:auth`,
  - `data:auth`,
  - `server`,
  - `shared`,
  - `core/common`.
- Модули для organizer/ticketing/lineup/donations, outbox/workers, realtime, payments и storage как отдельные bounded contexts пока не заведены.

### 06. Доменная модель и данные

- Реально существуют backend foundation-сущности и таблицы:
  - `users`,
  - `auth_identities`,
  - `user_role_assignments`,
  - `organizer_workspaces`,
  - `workspace_members`,
  - `refresh_tokens`,
  - `telegram_auth_assertions`.
- Полная доменная модель из документа пока не реализована.
- Особенно важно: `AuthIdentity`, `UserRoleAssignment`, `OrganizerWorkspace`, `Venue`, `Event`, `InventoryUnit`, `TicketOrder`, `DonationIntent` и связанные инварианты ещё отсутствуют в коде.

### 07. Интеграции и платежи

- Telegram flow интегрирован через backend verify + mobile callback bridge.
- VK/Google пока только scaffolded.
- `Sign in with Apple` отсутствует.
- PSP, donation flow, payout verification и legal settlement model в коде не реализованы.

### 08. API и событийная модель

- Фактический API surface ограничен:
  - `GET /health`,
  - `GET /auth/telegram/callback`,
  - `POST /api/v1/auth/telegram/verify`,
  - `GET /api/v1/auth/session/me`,
  - `POST /api/v1/auth/refresh`,
  - `POST /api/v1/auth/logout`.
- `me/roles`, `me/active-role` и `workspaces` endpoints уже существуют.
- Event/ticketing/donation/webhook/websocket endpoints пока не существуют.
- Внутренний event bus/outbox пока не внедрён.

### 09. NFR, безопасность и риски

- Для auth/session foundation уже реализованы:
  - secure storage,
  - request IDs,
  - body-size limits,
  - rate limiting,
  - replay protection,
  - refresh-token rotation,
  - structured auth logging.
- NFR для ticketing, payouts, live event delivery и check-in остаются целевыми требованиями для будущих доменов.

### 10. Дорожная карта и план релиза

- Репозиторий находится внутри `Phase 1`, но ещё не завершил её.
- Auth/session foundation собран.
- Backend foundation для roles/workspaces начат, но client и оставшиеся product domains ещё впереди.

## Рекомендуемый следующий шаг

- Следующий логичный delivery slice: закрыть оставшийся `Phase 1` foundation без расширения набора реально работающих провайдеров на этом шаге.
- Минимальный состав:
  - unified provider-agnostic user identity,
  - `AuthIdentity` model, где Telegram выступает первым linked provider,
  - role assignments + active role context,
  - organizer workspace membership model,
  - отвязка session/profile/RBAC от `telegram_id`.

## Важное ограничение текущей фазы

- Telegram может оставаться единственным реально работающим login provider временно.
- Это не меняет финальный MVP scope: `VK`, `Google` и `Sign in with Apple` всё ещё обязательны до публичного релиза.
- Поэтому следующая реализация должна готовить внутреннюю identity-модель под несколько провайдеров, а не закреплять Telegram как постоянный источник истины о пользователе.
