# 03. Роли и сценарии

## 3.1. Модель аккаунта

Один аккаунт хранит:
- identity;
- набор доступных ролей;
- профиль;
- историю покупок;
- историю участий как комика;
- участие в organizer workspaces;
- payout/verification status.

### Важно

Роль — это не отдельный аккаунт, а набор прав и стартовый домашний сценарий.

Пример:
- пользователь может купить билет как зритель;
- затем подать заявку как комик;
- затем получить приглашение в команду организатора как checker.

## 3.2. Основные роли и подроли

### Базовые продуктовые роли

- `Audience`
- `Comedian`
- `Organizer`

### Операционные подроли внутри organizer workspace

- `Owner` — полный контроль над организацией/событиями/финансами.
- `Manager` — редактирует событие, лайнап, гостей, продажи.
- `Checker` — check-in на входе, просмотр билетов, без доступа к финансам.
- `Host/Coordinator` — live-управление лайнапом и сценой, без финансовых действий.

## 3.4. Ключевые сценарии зрителя

### Сценарий A1. Найти и купить билет

1. Пользователь открывает каталог мероприятий.
2. Фильтрует по городу, дате, площадке, цене.
3. Открывает карточку события.
4. Просматривает лайнап, описание, схему зала, наличие мест.
5. Выбирает место или тариф.
6. Создаёт hold на время оплаты.
7. Оплачивает билет.
8. Получает QR-билет и уведомление.

### Сценарий A2. Прийти на мероприятие

1. Открывает “Мои билеты”.
2. Показывает QR на входе.
3. Получает статус checked-in.
4. Во время шоу смотрит live-статус сцены.

### Сценарий A3. Отправить донат комику

1. Видит текущего комика или открывает список участников.
2. Выбирает сумму из пресетов или вводит вручную.
3. По желанию добавляет короткое сообщение.
4. Оплачивает донат.
5. Видит подтверждение.

## 3.5. Ключевые сценарии комика

### Сценарий C1. Подать заявку на мероприятие

1. Комик открывает карточку события.
2. Нажимает “Подать заявку”.
3. Прикладывает базовые данные:
   - короткое описание;
   - длительность сета;
   - теги/формат;
   - при необходимости ссылки/медиа.
4. Получает статус `submitted`.

### Сценарий C2. Получить подтверждение и увидеть порядок

1. Организатор переводит заявку в `approved`.
2. Комик получает push.
3. Видит:
   - подтверждение участия;
   - время сбора;
   - ориентировочный слот;
   - текущий порядок выступлений.

### Сценарий C3. Live-режим выступления

1. Во время мероприятия комик открывает экран события.
2. Видит:
   - кто сейчас на сцене;
   - кто следующий;
   - свой слот;
   - если порядок поменялся — обновление в реальном времени.

## 3.6. Ключевые сценарии организатора

### Сценарий O1. Создать площадку и шаблон зала

1. Организатор создаёт площадку.
2. Указывает адрес, описание, ограничения и контакты.
3. Собирает схему зала:
   - сцена;
   - сектора;
   - ряды;
   - столы;
   - цены по зонам;
   - служебные/недоступные зоны.
4. Сохраняет как template.

### Сценарий O2. Создать мероприятие

1. Выбирает площадку/template.
2. Указывает дату, время, формат, описание.
3. Настраивает продажи:
   - старт/стоп продаж;
   - цены;
   - квоты;
   - комплименты;
   - промокоды;
   - waitlist policy.
4. Публикует событие.

### Сценарий O3. Собрать лайнап

1. Организатор просматривает заявки.
2. Shortlist/approve/reject.
3. Формирует порядок.
4. Назначает ответственного за сцену.
5. Во время события меняет фактический порядок и текущего комика.

### Сценарий O4. Провести мероприятие

1. Checker делает check-in гостей.
2. Host двигает stage status.
3. Организатор следит за:
   - продажами;
   - списком гостей;
   - лайнапом;
   - донатами;
   - инцидентами.

## 3.7. Жизненный цикл мероприятия

| Статус | Описание |
| --- | --- |
| `draft` | Черновик, не виден публике |
| `published` | Публичная карточка доступна |
| `sales_open` | Продажи открыты |
| `sales_paused` | Пауза продаж |
| `sold_out` | Нет доступного инвентаря |
| `in_progress` | Мероприятие идёт |
| `completed` | Мероприятие завершено |
| `canceled` | Мероприятие отменено |
| `archived` | Архив, read-only |

## 3.8. Жизненный цикл заявки комика

| Статус | Описание |
| --- | --- |
| `submitted` | Подана |
| `shortlisted` | В шорт-листе |
| `approved` | Подтверждена |
| `waitlisted` | Резерв |
| `rejected` | Отклонена |
| `withdrawn` | Комик снял заявку сам |

## 3.9. Жизненный цикл места/билета

| Статус | Описание |
| --- | --- |
| `available` | Доступно для продажи |
| `held` | Забронировано на время оплаты |
| `sold` | Оплачено |
| `checked_in` | Погашено на входе |
| `refunded` | Возврат выполнен |
| `blocked` | Не продаётся |
| `complimentary` | Выдано без оплаты |

## 3.10. Сценарии, которые обязательно нужно закрыть до запуска

- Переключение ролей внутри одного аккаунта.
- Приглашение сотрудника в organizer workspace.
- Hold места и защита от double booking.
- Live-перестановка порядка выступлений.
- Check-in без полного доступа к финансовым данным.
- Возврат места в продажу после отмены/истечения hold.
- Донат текущему комику за 1-2 действия.
