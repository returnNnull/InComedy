# 07. Интеграции и платежи

## 7.1. Авторизация: целевая стратегия

### Поддерживаемые провайдеры

- Telegram
- VK
- Google
- Apple (`Sign in with Apple`) для iOS

## 7.2. Рекомендуемый auth-подход

### Telegram

Рекомендуемый сценарий:
- запуск browser-based flow;
- получение callback/deeplink;
- server-side verification auth payload;
- обмен на внутреннюю сессию платформы.

Это соответствует уже существующему заделу в проекте и не завязывает продукт на нестабильный custom native flow.

### VK

Рекомендуемый сценарий:
- OAuth/OIDC-подобный flow через system browser + PKCE;
- backend exchange code -> internal session;
- возможность позже добавить native SDK wrapper без изменения domain contract.

### Google

Для лучшей конверсии лучше предусмотреть native wrappers:
- Android: современный Google Identity / Credential Manager path;
- iOS: native Google Sign-In wrapper.

Но серверная модель аккаунта должна оставаться единой.

### Apple

Для публикации iOS-приложения это не “опция”, а обязательный capability, если остаются сторонние social logins.

## 7.3. App Store / Google Play ограничения

### iOS

Нужно учитывать два отдельных ограничения:

1. Если приложение использует сторонние логины, App Store обычно требует `Sign in with Apple`.
2. Билеты на офлайн-мероприятия относятся к реальным услугам и могут оплачиваться внешним PSP, но donation flow надо проектировать осторожно.

### Практическое решение для iOS

- Билеты: внешний merchant/PSP flow допустим.
- Донаты:
  - вариант A: pass-through платеж, где вся сумма идёт комику;
  - вариант B: web checkout вне in-app digital goods модели.

Для MVP safest choice:
- не брать platform commission с донатов внутри iOS-приложения;
- держать donation payout как отдельную юридически чистую схему;
- если есть сомнение в App Review трактовке, делать donation checkout через web.

### Android / Google Play

- Для билетов на реальные офлайн-мероприятия store billing не обязателен.
- Для российского рынка важно учитывать ограничения Google Play Billing для продавцов с российским банком.

Вывод: билеты и донаты должны опираться на внешний PSP, а не на store billing.

## 7.4. Рекомендация по payment gateway

### Приоритетный кандидат для российского рынка

`YooKassa`

Причины:
- локальный рынок;
- зрелый эквайринг;
- сценарии marketplace/split settlement;
- понятная интеграционная документация;
- готовые mobile/web SDK сценарии.

### Резервный кандидат

`CloudPayments`

Полезен, если:
- уже есть merchant contract;
- нужна альтернативная acquiring-модель;
- важны отдельные особенности checkout/кассы.

### Архитектурное правило

Нельзя зашивать конкретный PSP в domain.

Нужен слой `PaymentGateway` / `PayoutGateway`, который абстрагирует:
- payment intent creation;
- payment status polling;
- webhook verification;
- refund;
- payout status.

## 7.5. Платёжная модель билета

### Ticket purchase flow

1. Клиент запрашивает hold мест.
2. Backend создаёт `TicketOrder` со статусом `awaiting_payment`.
3. Backend создаёт payment session в PSP.
4. Клиент завершает оплату.
5. PSP шлёт webhook.
6. Backend подтверждает оплату, выпускает билеты и QR.
7. Уведомления отправляются пользователю и организатору.

### Обязательные правила

- idempotency key на создание платежа;
- верификация webhook signature;
- повторная обработка webhook должна быть безопасной;
- ticket issuance только после подтверждённой оплаты;
- refund тоже должен быть идемпотентен.

## 7.6. Платёжная модель доната

Донат — это не разновидность билета, а отдельный домен.

### Donation flow

1. Зритель выбирает комика.
2. Вводит сумму.
3. Backend создаёт `DonationIntent`.
4. Запускается отдельная PSP payment session.
5. После webhook статус меняется на `paid`.
6. Система отражает донат:
   - у зрителя;
   - у комика;
   - в payout queue.

### Обязательные ограничения

- донат только verified comedian;
- лимиты по сумме и частоте;
- антифрод и velocity checks;
- блокировка донатов при payout/compliance issues;
- недопустимо смешивать donation funds и organizer revenue без явной юридической схемы.

## 7.7. Юридическая и финансовая модель

Главный риск не технический, а операционно-юридический:

- кому юридически приходит донат;
- кто платит комиссию;
- кто отвечает за чеки/налоги;
- можно ли выводить деньги физлицу, самозанятому или ИП;
- как возвращать ошибочные платежи.

### Рекомендация для MVP

- билетная выручка идёт организатору/платформе по отдельной merchant-схеме;
- донаты доступны только для комиков с верифицированным payout profile;
- если payout automation не готова, использовать `manual settlement with ledger`;
- явно фиксировать комиссию и ответственность в оферте.

## 7.8. Push и сервисные интеграции

Нужны интеграции:
- APNs;
- FCM;
- почта/служебные уведомления;
- аналитика;
- Sentry/ошибки;
- QR scanner/camera layer на клиентах.

## 7.9. Что обязательно предусмотреть заранее

- universal/deep links для карточек событий;
- callback domain strategy;
- привязка user identity к нескольким provider identities;
- миграция auth providers без потери аккаунта;
- повторная попытка оплаты;
- recovery flow для заказов в статусе “деньги списаны, билет не выдан”.
