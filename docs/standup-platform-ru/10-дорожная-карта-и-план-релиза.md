# 10. Дорожная карта и план релиза

## 10.1. Рекомендуемая стратегия

Не пытаться выпускать “полный стендап-superapp” за один релиз. Правильнее идти слоями:

1. foundation;
2. organizer ops;
3. ticketing;
4. live show layer;
5. donations;
6. growth/analytics.

## 10.2. Фазы

## Phase 0. Product foundation

Цель:
- утвердить scope;
- утвердить store-compliance;
- зафиксировать payment/payout model;
- согласовать сущности и статусы.

Результат:
- текущий пакет документации;
- утверждённый backlog;
- список интеграционных решений.

## Phase 1. Identity and role foundation

Входит:
- единый user identity;
- Telegram/VK/Google auth;
- `Sign in with Apple`;
- role onboarding;
- profile;
- workspace membership model.

Exit criteria:
- пользователь входит, восстанавливает сессию, переключает роли;
- organizer может пригласить сотрудника;
- permission model покрыт тестами.

## Phase 2. Organizer core and venue builder

Входит:
- organizer dashboard;
- venues;
- hall template builder v1;
- events;
- staff assignments;
- event publish/sales statuses.

Exit criteria:
- organizer создаёт площадку;
- собирает схему зала;
- публикует событие и открывает продажи.

## Phase 3. Ticketing and check-in

Входит:
- inventory;
- holds;
- orders;
- payment integration;
- QR tickets;
- check-in;
- refund/cancel baseline;
- waitlist.

Exit criteria:
- зритель покупает билет;
- получает QR;
- checker валидирует билет;
- refunded seat возвращается в продажу.

## Phase 4. Comedian and live show operations

Входит:
- comedian profile;
- applications;
- lineup;
- live stage status;
- event feed/announcements;
- push notifications.

Exit criteria:
- комик подаёт заявку;
- организатор формирует лайнап;
- зритель и комик видят текущего выступающего.

## Phase 5. Donations and stabilization

Входит:
- donation intent/payment;
- payout profile verification;
- comedian donations history;
- operational admin tooling;
- observability hardening.

Exit criteria:
- донат проходит end-to-end;
- payout pipeline прозрачен;
- инциденты по платежам диагностируются по логам и dashboard.

## 10.3. Рекомендуемый MVP-состав

Если нужно сократить объём, MVP должен включать:
- auth + roles;
- organizer workspace;
- venue builder v1;
- event management;
- ticketing + check-in;
- comedian applications + lineup;
- live current performer;
- donations v1;
- push.

Что можно подрезать первым:
- публичный чат;
- сложную аналитику;
- рекомендации;
- автоматизацию сложных выплат;
- расширенный профиль комика.

## 10.4. Рекомендуемый состав команды

Минимально жизнеспособная команда:
- 1 product manager / analyst;
- 1 backend engineer;
- 1 KMP/shared engineer;
- 1 Android/iOS engineer с уклоном в platform wrappers;
- 1 QA automation / manual hybrid;
- 1 product designer part-time.

## 10.6. MVP exit criteria

MVP готов к beta-запуску, если:
- организатор создаёт событие и открывает продажи без ручных таблиц;
- зритель покупает билет и проходит check-in;
- комик подаёт заявку и видит live-статус шоу;
- донат проходит от зрителя к комику через поддерживаемую операционную схему;
- критичные инциденты диагностируются без доступа к прод-базе вручную.

## 10.7. Основные зависимости

- согласование юрсхемы по донатам и выплатам;
- выбор PSP;
- подтверждение auth flows для App Store;
- определение формата venue templates;
- базовая QA-автоматизация на ticketing.

## 10.8. Самые опасные ошибки планирования

- запускать публичный чат раньше стабильного ticketing;
- тянуть в MVP сложную социальную ленту;
- не заложить staff roles и audit log;
- не зафиксировать refund policy до релиза;
- недооценить store-compliance по auth и donations.
