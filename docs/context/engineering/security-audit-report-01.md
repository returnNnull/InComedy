# Security Audit Report: :server

Дата: `2026-03-21`
Статус: `Draft / Recommendations`

## 1. Резюме
Модуль `:server` демонстрирует зрелый подход к безопасности (использование JWT, ротация Refresh Token, защита от SQL-инъекций, Rate Limiting). Однако для достижения максимального уровня безопасности по современным стандартам (OWASP ASVS, NIST) требуется внедрение системных механизмов проверки отзыва токенов и усиление транспортной безопасности.

## 2. Детальный анализ

### 2.1 Аутентификация и сессии
- **[OK]** Ротация Refresh Token (one-time use).
- **[OK]** Хеширование Refresh Token в БД (SHA-256).
- **[OK]** Защита от timing-атак при валидации Telegram hash.
- **[RISK]** **Отсутствие проверки отзыва Access Token:** Токен валиден до истечения срока (exp), даже если пользователь вышел из системы или сессия была отозвана в БД.
- **[REC]** Внедрить Ktor-плагин для сверки `iat` (время выпуска) с `session_revoked_at` пользователя при каждом запросе.

### 2.2 Транспортная безопасность
- **[OK]** Принудительный SSL для PostgreSQL и Redis при удаленных подключениях.
- **[RISK]** **Отсутствие Security Headers:** Не установлены HSTS, X-Content-Type-Options, CSP, X-Frame-Options.
- **[REC]** Настроить плагин `DefaultHeaders` для установки базовых защитных заголовков.

### 2.3 Защита от перебора и DoS
- **[OK]** Rate Limiting на базе Redis/In-memory для эндпоинтов авторизации.
- **[RISK]** **Подмена IP:** Текущая логика `X-Forwarded-For` доверяет любому входящему заголовку.
- **[REC]** Использовать `ForwardedHeaderSupport` с белым списком доверенных прокси.

### 2.4 Архитектура и Инфраструктура
- **[OK]** Secrets Management через Environment Variables.
- **[OK]** CALL-ID трассировка для всех запросов.
- **[REC]** Переход с HMAC256 на RS256 (асимметричное шифрование) для подписи JWT.

## 3. План действий (P0-P1)

1. **[P0]** Реализовать `RevocationCheckPlugin` для валидации статуса сессии в БД.
2. **[P0]** Настроить `DefaultHeaders` (Security Headers).
3. **[P1]** Настроить `ForwardedHeaderSupport` для безопасного извлечения IP.
4. **[P1]** Интегрировать `Sanitized Diagnostics` для логирования критических этапов безопасности согласно стандартам проекта.
